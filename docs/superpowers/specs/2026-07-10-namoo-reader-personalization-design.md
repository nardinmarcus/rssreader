# Namoo Reader 个性化设计

**日期：** 2026-07-10
**状态：** 已通过对话确认，待用户复核书面规范

## 1. 背景

当前项目继承自 QMReader，产品文案、信息源、管理员名称、统计脚本、RSSHub 配置、创作提示词、公开资源和部署命名中仍包含大量乔木个人信息。项目已经以单容器形式部署到 `https://rss.namooca.com`，下一步需要在不破坏现有阅读数据和部署结构的前提下，将其改造成服务于大月 Namoo 的 AI 阅读与创作工作流。

## 2. 目标

1. 将产品统一命名为 **Namoo Reader**，管理员默认名称改为 **大月 Namoo**。
2. 保留现有米白浅色主题、深色主题和页面布局，只替换品牌文字与图标。
3. 保留现有 61 个信息源，同时加入 Namoo 的核心 AI 与创作信息源。
4. 为信息源增加启用开关、编辑优先级、标签和可持久化的侧边栏顺序。
5. 将“乔木风格改写”重构为“Namoo 创作草稿”，服务于从阅读、提炼到人工创作的流程。
6. 保持单应用容器，不增加独立 RSSHub 容器。
7. 全面清理运行路径中的乔木品牌残留，同时保留合法的上游版权和历史说明。
8. 完成自动化验证、生产部署和 `https://rss.namooca.com` 端到端验收。

## 3. 非目标

- 不重做浅色或深色主题，不调整主要页面布局。
- 不加入微信公众号、博客、X 等自动发布能力。
- 不自建 RSSHub，也不增加第二个应用容器。
- 不删除已有文章、阅读历史或被关闭信息源的历史数据。
- 不删除或改写上游 MIT 版权归属。
- 本阶段不提供标签编辑器；标签由信息源目录定义，后台可按标签筛选。

## 4. 品牌系统

### 4.1 品牌内容

- 产品名：`Namoo Reader`
- 包名与容器服务名：`namoo-reader`
- 管理员默认名：`大月 Namoo`
- 生产站点：`https://rss.namooca.com`
- 仓库地址：`nardinmarcus/rssreader`
- 产品定位：面向 AI 内容阅读与创作转化的个人信息站。
- 创作表达：有见识的普通人在认真聊一件打动他的事。

站点标题、描述、页面品牌、空状态、RSS、Open Graph、JSON-LD、`llms.txt`、安装图标和分享元数据必须使用同一套品牌信息，避免页面、搜索引擎和订阅输出之间出现不同名称。

### 4.2 图标

采用已确认的 B 方案“叶片负形 N”：以简洁叶片轮廓作为外形，以负形构成字母 N。图标必须：

- 使用单色几何结构，不依赖渐变、阴影或细小装饰；
- 在 16px、32px、180px 和 512px 尺寸下保持可辨认；
- 同时适配米白浅色背景和深色背景；
- 输出一个主 SVG，并派生 favicon 和移动端图标；
- 不改变现有页面配色和布局。

## 5. 信息源模型

### 5.1 静态目录与个人状态

`lib/sources.js` 继续作为可用信息源的版本化目录。每个信息源需要稳定且不可随标题变化的 `id`，并包含名称、抓取地址、标签、默认启用状态、默认编辑优先级、默认展示顺序和原有抓取优先级。

SQLite 新增 `source_preferences` 表，用于保存管理员通过界面产生的个人状态：

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `source_id` | TEXT | 主键，对应静态目录中的稳定 ID |
| `enabled` | INTEGER | `0` 或 `1` |
| `editorial_priority` | TEXT | `high`、`normal` 或 `low` |
| `display_order` | INTEGER | 全局侧边栏顺序，值越小越靠前 |
| `updated_at` | TEXT | ISO 8601 时间 |

启动时按 `source_id` 合并两层数据：存在个人设置时优先使用 SQLite；没有设置时使用静态目录默认值。SQLite 是持久化用户数据，不是运行时缓存。静态目录新增信息源后，无需数据库迁移即可按默认值出现。

标签在本阶段属于静态目录元数据，可包含多个值。初始标签集合为：`官方`、`研究`、`产品`、`产业`、`社区`、`创作`、`上游来源`。后台支持筛选，但本阶段不增加标签编辑器。

### 5.2 优先级分离

- `refreshPriority`：保留现有含义，只决定抓取调度频率。
- `editorialPriority`：新增内容价值优先级，分为高、中、低，用于后台筛选、标识和人工选题判断。
- 两者互不覆盖，避免“更新频繁”被误解为“更值得阅读”。

默认文章列表继续按发布时间排序；编辑优先级本身不改变文章时间顺序。本阶段不增加隐藏的推荐算法或加权信息流。

### 5.3 启用规则

关闭一个信息源后：

1. 后续抓取任务跳过该源；
2. 该源不出现在普通侧边栏和默认信息流中；
3. 管理后台仍显示该源，并可按“已关闭”筛选；
4. 已抓取文章继续保存在 SQLite；
5. 文章深层链接和历史记录仍可访问；
6. 重新启用后恢复抓取，不重复清空或重建历史文章。

乔木博客保留在目录中，默认关闭，并标记为 `上游来源`。

### 5.4 排序规则

管理员登录后，可在左侧栏对单个信息源执行“上移”或“下移”。普通访问者不显示排序控件。

每次移动在一个 SQLite 事务中交换相邻两项的 `display_order`。服务端负责处理首项上移、末项下移、缺失顺序和重复顺序；必要时按当前有效顺序重新编号。排序是全局管理员顺序，不按设备或浏览器分别保存。

### 5.5 Namoo 核心源

保留现有 61 个信息源，并在实施阶段验证后补入一组默认启用的 Namoo 核心源。核心组至少覆盖以下来源或其官方等价订阅：

- 官方实验室：OpenAI、Anthropic、Google DeepMind、Meta AI、Hugging Face；
- 研究与工程解读：Import AI、The Batch、Simon Willison；
- AI 产品与产业观察：Latent Space 以及经过验证的高质量产品动态源；
- 创作方法：与个人写作、观点构建和内容表达直接相关的高质量来源。

新增源必须逐个通过真实订阅地址、最近更新时间和抓取结果验证。没有可用直连 RSS 时才使用 RSSHub；验证失败的候选源可以进入目录，但必须默认关闭并记录原因，不能持续制造后台错误。

## 6. RSSHub 与统计配置

### 6.1 RSSHub

新增逗号分隔的 `RSSHUB_INSTANCES` 环境变量。未配置时使用现有三个公共实例作为回退。应用按列表顺序尝试实例；一个实例失败后继续尝试下一个，所有实例失败时记录该信息源的明确错误。

RSSHub 继续作为远程依赖使用。生产部署仍只有现有 Namoo Reader 应用容器，不创建 RSSHub 容器。

### 6.2 Umami

`UMAMI_SRC` 和 `UMAMI_WEBSITE_ID` 默认都为空。只有两个值同时存在时才向页面注入统计脚本；任一缺失都不加载脚本。仓库和生产默认配置不得继续引用乔木的 Umami 地址或网站 ID。

## 7. Namoo 创作草稿

### 7.1 用户流程

阅读原文后，用户可执行“生成 Namoo 创作草稿”：

1. 提取原文事实、数字、人物、时间和链接；
2. 说明该内容为什么值得写；
3. 给出 2 至 3 个创作角度和标题方向；
4. 根据选定内容生成 Namoo 风格的结构化草稿；
5. 显式标出必须由 Namoo 补充的真人内容；
6. 给出发布前事实核查清单。

### 7.2 输出结构

输出固定包含：

1. `为什么值得写`
2. `创作角度`
3. `事实底稿与原始链接`
4. `Namoo 风格草稿`
5. `需要 Namoo 补充`
6. `发布前检查`

需要真实经验、测试、情绪或个人判断的位置使用清晰占位，例如：`[需要 Namoo 补充：你实际试用该产品后的感受]`。模型不得伪造第一人称经历、实际测试或情绪。

### 7.3 失败与安全边界

- AI 密钥只保存在服务端，不进入浏览器响应或日志正文；
- 生成失败不影响原文阅读，用户可以单独重试；
- 原文、生成草稿和来源链接保持关联；
- 提示词和界面不得继续使用“乔木风格改写”；
- 保留现有 AI 服务端配置兼容性，本阶段不额外抽象多供应商系统；
- 草稿必须经过人工补充和审核，不提供自动发布。

## 8. 运行路径全面审计

实施时检查以下范围：

- `public/` 中页面品牌、图标、manifest 和客户端文案；
- `server.js` 中 SEO、RSS、结构化数据、资源元数据和默认值；
- `lib/sources.js` 中个人来源、描述和 RSSHub 配置；
- `lib/deepseek.js` 与客户端中的创作提示词和结果标签；
- `.env.example`、Docker Compose 和生产环境示例；
- `package.json`、README、贡献、安全及行为准则文档；
- 容器、服务、日志和健康检查中的产品命名；
- `ops/` 历史部署记录。

验收搜索词包括：`乔木`、`向阳乔木`、`Qiaomu`、`QMReader`、`qiaomu.ai`、乔木的 Umami 地址及网站 ID。只有以下位置允许保留相关内容：

1. 上游 MIT 版权和来源归属；
2. 明确标记为“上游历史资料”的 `ops/` 记录；
3. 迁移说明中为了说明旧名称而出现的一次性文字。

`ops/` 不伪装为 Namoo 的历史记录。它将通过索引说明或独立目录与当前生产运维文档隔离。

## 9. API 与界面行为

- 普通信息源接口默认只返回已启用源；管理员接口可以返回全部源及其偏好状态。
- 已通过现有管理员会话认证的请求可更新启用状态和编辑优先级，并执行相邻上移或下移；未认证请求只能读取公开状态。
- 服务端验证 `source_id`、优先级枚举和移动方向，不接受未知源或任意 SQL 排序值。
- 修改成功后返回合并后的最新源状态；客户端使用服务端结果刷新，不自行猜测最终顺序。
- 管理后台可按标签、编辑优先级、启用状态和最近错误状态筛选。
- 左侧栏沿用现有样式，只增加紧凑的管理员操作控件，不改变普通阅读布局。

## 10. 数据迁移与回滚

数据库迁移必须可重复执行：表或索引已存在时不报错，不修改已有文章表和阅读数据。部署前备份生产 SQLite 与环境配置。

回滚时恢复旧镜像和旧环境配置即可继续读取原有文章；新增 `source_preferences` 表可以留在数据库中，因为旧版本不会读取它。若新版本运行异常，不以删除数据库作为恢复手段。

## 11. 测试与验收

### 11.1 自动化测试

- 静态目录与 SQLite 偏好的合并优先级；
- 新增源在无个人设置时采用默认值；
- 关闭源后抓取任务跳过，普通接口隐藏，管理员接口仍显示；
- `refreshPriority` 与 `editorialPriority` 相互独立；
- 首项、末项和中间项的上下移动；
- 重复或缺失排序值的规范化；
- 数据库迁移重复执行；
- Umami 两项配置不完整时不注入脚本；
- RSSHub 多实例依次回退与全部失败错误；
- 创作提示词包含事实引用、真人占位和禁止编造要求。

### 11.2 浏览器与生产验收

- 米白浅色和深色主题均保持原布局；
- favicon 在浏览器标签和移动图标尺寸下可辨认；
- 管理员可启停、筛选、上移和下移信息源；
- 刷新页面和重启容器后顺序、优先级与启用状态保持；
- 被关闭源不再抓取且不出现在默认阅读界面，历史文章仍可访问；
- Namoo 创作草稿结构完整，失败可重试，原文不受影响；
- 首页、RSS、SEO 元数据、管理员登录、信息源接口和后台任务正常；
- 生产仍只有一个 Namoo Reader 应用容器；
- `https://rss.namooca.com` 返回 HTTP 200，运行日志没有启动级错误。

## 12. 部署完成标准

1. 备份生产数据库和环境配置；
2. 构建并验证新镜像；
3. 在原部署目录替换现有单容器；
4. 执行数据库迁移和健康检查；
5. 验证公开站点、后台、抓取、排序持久化和创作草稿；
6. 完成品牌残留扫描；
7. 只有全部验证通过，才将本次个性化标记为部署成功。
