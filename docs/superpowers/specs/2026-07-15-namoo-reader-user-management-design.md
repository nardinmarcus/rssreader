# Namoo Reader 用户管理设计

**日期：** 2026-07-15
**状态：** 用户已批准书面规范

## 1. 背景

Namoo Reader 已支持站点注册、登录、个人空间、用户投稿以及管理员审核。当前管理员可以在“我的空间 → 站点管理 → 内容审核 → 投稿账号”查看账号并执行投稿查看、账号停用、账号恢复和投稿下线。

现有入口把账号治理混在内容审核中，名称也暗示只展示投稿者，但实际接口读取全部注册用户。管理员难以发现入口，也缺少面向账号治理的搜索、筛选、分页、状态详情和长期操作记录。

本设计将用户管理从内容审核中拆出，继续复用“我的空间”的页面容器、管理员会话和 SQLite 数据模型，不建立第二套后台。

## 2. 目标

1. 为管理员提供清晰、独立的“用户管理”入口。
2. 查看全部注册用户，而不是只查看有投稿记录的用户。
3. 支持按显示名或邮箱搜索，并按状态、角色筛选和分页。
4. 在同一页面查看用户身份、账号状态、注册与登录时间、投稿数量和最近投稿。
5. 安全地执行投稿下线、账号停用和账号恢复。
6. 保留不可覆盖的管理员操作记录。
7. 保证停用账号涉及的状态变化要么一起提交，要么一起回滚。
8. 保持 SQLite 为持久化事实来源，不把运行时缓存或浏览器状态当作账号真值。

## 3. 非目标

- 不允许管理员修改用户邮箱、显示名、头像、角色或密码。
- 不提供硬删除账号。
- 不提供批量停用、批量恢复或批量删除投稿。
- 不增加新的管理员角色或细粒度权限系统。
- 不建立独立的后台应用、后台页面壳或第二套餐单。
- 不恢复已被下线的投稿，也不恢复失效的旧会话。
- 不增加全站用户增长分析、留存分析或运营报表。
- 不改造普通用户的个人资料和账户安全流程。

## 4. 已确认的产品决策

### 4.1 信息架构

“我的空间 → 站点管理”调整为以下顺序：

1. 用户管理
2. 内容审核
3. 订阅源

“内容审核”只保留待审投稿。原来的“投稿账号”列表移动到独立的“用户管理”标签，并扩展为全部注册用户列表。

### 4.2 页面布局

桌面端采用主从双栏：

- 左侧是可搜索、筛选、排序和分页的用户列表；
- 右侧是当前选中用户的详情、最近投稿、操作记录和治理动作；
- 选择另一个用户时，列表上下文保持可见。

移动端不压缩双栏。选择用户后进入独立详情视图，返回时恢复原列表的查询条件、页码和滚动位置。

### 4.3 停用确认

停用账号采用“原因 + 影响确认”：

1. 弹窗显示目标账号。
2. 显示将撤销的活跃会话数、将拒绝的待审投稿数和将下线的公开投稿数。
3. 管理员必须填写停用原因。
4. 管理员必须勾选“已确认账号和受影响数量”。
5. 最后才能提交停用操作。

不要求管理员再次输入用户邮箱。当前没有批量操作，且管理员账号受服务端保护，额外输入邮箱只会增加操作负担。

## 5. 路由与导航

### 5.1 主路由

- 用户管理入口：/me?tab=users
- 选中用户：/me?tab=users&user={user-id}
- 搜索、筛选、排序和页码继续写入查询参数，保证刷新、前进和后退可恢复。

完整状态示例：

    /me?tab=users&user={user-id}&q=lin&status=active&role=user&sort=last_login_desc&page=1

### 5.2 管理员兼容入口

/admin 默认进入用户管理，并替换为规范路由 /me?tab=users。订阅源仍可通过 /me?tab=sources 直接访问。

### 5.3 非管理员访问

- 未登录用户访问管理路由时打开登录入口。
- 已登录普通用户访问时回到 /me?tab=profile，并显示“需要管理员权限”。
- 前端隐藏管理标签只是体验处理；所有接口仍由服务端权限校验。

### 5.4 与内容审核的联动

待审投稿中的投稿人名称可跳转到对应用户详情。跳转时保留返回“内容审核”的浏览器历史。

## 6. 用户列表

### 6.1 列表字段

每行显示：

- 头像
- 显示名
- 邮箱
- 角色：普通用户或管理员
- 状态：正常或已停用
- 注册时间
- 最近登录时间
- 公开投稿数
- 累计投稿数

管理员邮箱只出现在管理员受保护页面，不进入公开贡献者接口或页面元数据。

### 6.2 搜索与筛选

- 搜索范围：显示名和完整邮箱。
- 搜索词去除首尾空格，最长 100 个字符。
- 状态筛选：全部、正常、已停用。
- 角色筛选：全部、普通用户、管理员。
- 排序：最近注册、最近登录。
- 默认排序：最近注册。
- 搜索输入停止 250 毫秒后触发，避免每次按键都产生并发请求。

### 6.3 分页

- 默认每页 50 人。
- 最大每页 100 人。
- 分页由服务端完成。
- 前端只保存当前页，不再一次读取最多 500 个用户。
- 响应同时返回过滤后的总数和全站账号摘要。
- 页码超过最后一页时，服务端返回规范化后的最后一页；没有结果时页码为 1。

全站摘要包含：

- 注册用户总数
- 正常账号数
- 已停用账号数
- 管理员账号数

桌面端在 URL 没有 user 参数且当前页有结果时，自动选中第一位用户，并通过 replaceState 写入规范 URL，不增加一条返回历史。移动端不自动进入详情，必须由管理员点击用户。

## 7. 用户详情

右侧详情包含四个区块。

### 7.1 身份与状态

- 头像、显示名、邮箱和角色
- 当前状态
- 注册时间
- 最近登录时间
- 停用时间、停用原因和执行管理员

管理员账号显示保护标识，不显示停用按钮。

### 7.2 内容影响摘要

- 活跃会话数
- 待审投稿数
- 公开投稿数
- 累计投稿数
- 已下线投稿数

这些数字用于展示和确认，但执行操作时必须由服务端在事务内重新计算。

活跃会话只统计 expires_at 大于当前时间的记录。停用事务先清理目标用户的过期会话，再比较和撤销有效会话；审计中的 revokedSessionCount 因此与管理员看到的有效会话数保持一致。

### 7.3 投稿记录

- 默认显示最近 10 条投稿。
- 每条显示标题、地址、发布时间和当前公开状态。
- “查看全部投稿”继续使用现有按用户读取投稿的接口。

### 7.4 操作记录

默认显示最近 20 条管理员操作：

- 操作类型
- 操作管理员
- 原因
- 实际影响数量
- 操作时间

第一阶段不增加独立的全站审计日志页面。

## 8. 治理动作

### 8.1 下线全部投稿

该操作只影响当前公开投稿，不影响账号登录。

- 必须显示将下线的准确数量。
- 必须填写原因。
- 必须二次确认。
- 使用软下线，不删除文章行和用户投稿关系。
- 写入管理员操作记录。

### 8.2 停用账号

停用账号在一个 SQLite 事务内完成：

1. 重新读取目标账号并验证仍然存在。
2. 拒绝停用当前管理员自己或任何管理员账号。
3. 重新计算活跃会话、待审投稿和公开投稿数量。
4. 比较管理员刚刚确认的影响数量。
5. 数量或账号状态发生变化时返回冲突，不执行部分修改。
6. 写入账号停用时间、执行管理员和原因。
7. 撤销目标账号的全部会话。
8. 拒绝目标账号的全部待审投稿。
9. 软下线目标账号的全部公开投稿。
10. 写入管理员操作记录。
11. 一起提交事务。

任何一步失败都回滚全部修改。

停用后：

- 现有会话立即失效；
- 新登录返回“账号已被管理员停用”；
- 该用户不能创建新投稿；
- 已下线投稿不再出现在公开列表；
- 历史数据和治理证据继续保留。

### 8.3 恢复账号

恢复账号只恢复登录资格：

- 清除当前停用状态；
- 不恢复旧会话；
- 不恢复待审投稿；
- 不恢复已下线投稿；
- 用户必须重新登录；
- 写入管理员操作记录。

恢复确认可以填写备注；空备注由服务端记录为“管理员恢复账号”。

### 8.4 幂等性

- 已停用账号再次收到停用请求时，先按目标状态判定为幂等成功，再返回当前状态；此时不比较旧的 expectedImpact，不重复下线内容或写入成功操作记录。
- 正常账号再次收到恢复请求时，先按目标状态判定为幂等成功，再返回当前状态；不重复写入成功操作记录。
- 客户端因网络中断重试时可以安全地重新读取状态。

### 8.5 公开投影刷新

停用账号或下线投稿提交成功后，抓取层使读者投稿投影失效，并从 SQLite 重建公开列表。该刷新发生在数据库事务提交之后，运行时缓存和磁盘缓存都不是操作是否成功的判据。

如果投影刷新失败：

- 已提交的账号和投稿状态不回滚；
- 接口返回数据库操作成功，并标记 projectionRefreshPending；
- 服务端记录刷新错误；
- 下一次公开读取必须先从 SQLite 重建投影，不能继续返回已下线条目。

这样可以避免数据库已经成功停用账号，但缓存写入失败却向管理员返回“整体失败”的不确定状态。

## 9. 数据模型

### 9.1 事实来源

users、sessions、submission_requests、user_submissions 和 entries 继续保存账号及治理状态。浏览器状态和任何运行时缓存只是读取投影，不参与权限判断或最终计数。

现有 users.disabled_at、users.disabled_by 和 users.disabled_reason 继续表达当前停用状态。

### 9.2 新增管理员操作记录

新增 admin_action_logs 表：

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| id | TEXT | 主键，UUID |
| actor_user_id | TEXT | 必填，执行管理员 |
| target_user_id | TEXT | 必填，目标用户 |
| action | TEXT | 必填，允许的动作枚举 |
| reason | TEXT | 必填，最长 300 字符 |
| impact_json | TEXT | 必填，实际影响数量的 JSON |
| created_at | INTEGER | 必填，毫秒时间戳 |

动作枚举：

- user.disable
- user.restore
- user.submissions_hide

impact_json 统一使用以下键：

- revokedSessionCount
- rejectedPendingCount
- hiddenSubmissionCount

没有影响的键保存为 0，避免读取端根据动作猜测结构。

新增索引：

- target_user_id + created_at
- actor_user_id + created_at

操作记录只在对应状态修改成功时写入，并与状态修改处于同一事务。失败尝试进入服务端错误日志，不写成成功审计记录。

## 10. 管理接口

### 10.1 用户列表

GET /api/admin/users

参数：

- q
- status：all、active、disabled
- role：all、user、admin
- sort：created_desc、last_login_desc
- page：默认 1
- limit：默认 50，最大 100

响应包含：

- users
- pagination：page、limit、filteredTotal、pageCount
- summary：total、active、disabled、admins

服务端拒绝未知枚举、负页码和超长搜索词，不接受客户端传入 SQL 排序表达式。

### 10.2 用户详情

GET /api/admin/users/:id

响应包含：

- user
- impact
- recentSubmissions
- recentActions

impact 是停用确认所需的当前数量快照。

### 10.3 全部投稿

GET /api/admin/users/:id/submissions

继续保留现有接口，并加入服务端分页。详情页只在管理员明确请求时加载完整列表。

### 10.4 停用

POST /api/admin/users/:id/disable

请求包含：

- confirmUserId
- reason
- expectedImpact

服务端不信任 expectedImpact，只把它用于和事务内真实计数比较。计数变化时返回 409 和新的 currentImpact，要求客户端重新展示确认内容。

现有 DELETE /api/admin/users/:id 暂时作为兼容入口，两条路由调用同一个处理函数。兼容入口同样必须提交 confirmUserId、reason 和 expectedImpact；缺少字段时返回 400，不以兼容为理由降低确认强度。第一阶段不删除兼容路由。

### 10.5 恢复

POST /api/admin/users/:id/restore

请求可以包含 reason。服务端仍需校验目标账号存在并记录实际执行结果。

### 10.6 投稿下线

DELETE /api/admin/users/:id/submissions

请求包含：

- confirmUserId
- reason
- expectedVisibleSubmissionCount

数量变化时返回 409，不在管理员未重新确认的情况下扩大影响范围。

## 11. 权限与隐私

- 用户管理接口按 requireLogin、requireAdmin 的顺序组合校验：未登录返回 401，已登录但不是管理员返回 403。现有全局 requireAdmin 的语义不在本任务中修改。
- 所有写接口继续使用现有同源和跨站写操作防护。
- 查询和写入都由服务端根据当前会话取得管理员身份，不接受客户端传入 actor_user_id。
- 不返回密码摘要、盐、会话令牌、AI 密钥或其他秘密。
- 普通用户不能通过公开贡献者接口读取其他用户邮箱、停用原因或操作记录。
- 服务端拒绝停用当前管理员和所有 role=admin 的账号。
- 第一阶段不支持改变角色，因此不存在通过页面提升权限的路径。

## 12. 并发与错误处理

### 12.1 并发变化

停用和投稿下线使用 BEGIN IMMEDIATE，在事务中重新查询账号状态和影响数量。

如果确认弹窗打开后出现新的会话、待审投稿或公开投稿，服务端返回 409，并返回当前影响数量。客户端更新弹窗，清除确认勾选，要求管理员再次确认。

### 12.2 状态码

- 400：参数、原因或确认数据无效。
- 401：会话失效，提示重新登录。
- 403：不是管理员，或尝试处置管理员账号。
- 404：目标用户不存在。
- 409：目标状态或影响数量已改变。
- 500：事务或服务端执行失败。

### 12.3 客户端行为

- 不对停用、恢复或投稿下线做乐观更新。
- 操作成功后重新读取当前用户详情和当前列表页。
- 收到 409 时保留管理员填写的原因，更新影响数量并要求重新确认。
- 收到 404 时关闭详情并刷新列表。
- 网络中断或 500 后保留页面上下文，重新读取状态后再判断是否需要重试。
- 列表读取失败不清空上一次成功结果；详情失败不破坏列表。

## 13. 组件边界

实施应沿用现有文件结构，只提取用户管理所需的局部边界：

- public/index.html：用户管理标签、列表容器、详情容器和确认弹窗结构。
- public/app.js：路由状态、列表查询、详情查询、渲染和治理动作协调。
- lib/store.js：分页查询、详情聚合、审计表和事务函数。
- server.js：管理员接口、参数验证和统一错误映射。
- test/admin-submissions.test.js 及相邻测试：权限、事务、迁移和 HTTP 回归。

不借此任务拆分整个 public/app.js、重写路由器或重构无关管理功能。

## 14. 数据迁移与回滚

### 14.1 迁移

- 使用现有加法式初始化路径创建 admin_action_logs 和索引。
- CREATE TABLE 和 CREATE INDEX 必须可重复执行。
- 不重写 users、entries 或现有投稿关系。
- 不修改任何现有用户密码摘要、角色、邮箱或停用状态。

### 14.2 部署前验证

在临时数据库和一份本地 SQLite 副本上重复执行两次初始化，验证：

- PRAGMA quick_check 返回 ok；
- 用户数、管理员数和文章数不变；
- 管理员密码摘要不变；
- 第二次初始化没有新增重复结构或数据。

### 14.3 回滚

旧版本不会读取 admin_action_logs。回滚应用镜像时可以保留该表，不需要删除或逆向迁移数据。

生产部署前备份 SQLite 和环境配置。数据库异常时通过一致性备份恢复，不通过删除数据库恢复。

## 15. 测试与验收

### 15.1 存储与迁移测试

- 审计表和索引可重复初始化。
- 用户分页、搜索、状态筛选、角色筛选和排序结果正确。
- 全站摘要不受当前页面限制影响。
- 用户详情的会话、待审投稿和投稿数量与真实表数据一致。
- 停用的所有修改和审计写入处于同一事务。
- 任一步失败时账号、会话、投稿和审计记录一起回滚。
- 恢复只清除当前停用状态。
- 重复停用和重复恢复保持幂等。
- 公开投影刷新失败不改变已提交的 SQLite 状态；下一次公开读取从 SQLite 重建且不返回已下线投稿。

### 15.2 API 测试

- 匿名和普通用户不能访问任何管理接口。
- 管理员可以读取列表和详情。
- 未知查询枚举、无效页码和超长搜索词返回 400。
- 管理员不能停用自己或其他管理员。
- expectedImpact 不一致返回 409，且没有产生部分修改。
- 停用成功返回实际影响数量并使旧会话失效。
- 恢复后可以创建新会话，但旧会话仍然失效。
- 投稿下线不影响账号登录。
- 兼容 DELETE 停用路由与新 POST 路由行为一致。
- 响应不包含密码和会话秘密。

### 15.3 浏览器测试

- 管理员可见“用户管理”，普通用户不可见。
- /admin 和 /me?tab=users 路由正确。
- 搜索、筛选、排序、分页和选择用户可以通过刷新、前进和后退恢复。
- 桌面端主从双栏保持列表上下文。
- 移动端详情返回后恢复列表位置。
- 停用弹窗要求原因和确认勾选。
- 409 会更新影响数字并清除旧确认。
- 恢复文案明确说明不会恢复旧会话和已下线内容。
- 内容审核中的投稿人可以跳转到用户详情。

### 15.4 生产验收

生产环境默认只做管理员只读验收，不制造测试账号或违规内容：

- 备份生产 SQLite 与环境配置。
- 数据库迁移后 PRAGMA quick_check 返回 ok。
- 用户总数、管理员数、文章数和管理员密码摘要符合迁移前快照。
- 管理员可以打开用户列表、搜索和读取详情。
- 未认证管理接口返回权限错误。
- 首页、/api/me、信息源、文章列表和管理员登录保持正常。
- 最近日志没有迁移、事务、权限或启动错误。

停用、恢复和投稿下线的破坏性链路由本地 HTTP 与浏览器端到端测试证明。只有出现真实治理需求时才在生产执行对应操作。

## 16. 实施顺序

1. 增加审计表、分页查询、用户详情聚合和事务级治理函数。
2. 补齐存储层与 API 回归测试。
3. 扩展管理员接口并保留旧停用路由兼容性。
4. 拆分“用户管理”和“内容审核”界面。
5. 实现桌面主从双栏、移动端详情和路由恢复。
6. 实现停用、恢复和投稿下线确认流程。
7. 完成浏览器回归、数据库副本迁移测试和生产只读验收。

每一步只修改实现该目标所需的代码，不借机重构无关页面或服务。

## 17. 完成标准

只有同时满足以下条件，用户管理第一阶段才算完成：

1. 管理员可以在独立入口查看全部注册用户。
2. 搜索、筛选、排序和分页结果以 SQLite 为准。
3. 用户详情和影响数字准确。
4. 停用账号事务原子且管理员账号受保护。
5. 恢复账号不会恢复旧会话和已下线内容。
6. 所有成功治理操作都有不可覆盖的操作记录。
7. 普通用户和匿名用户无法读取或修改管理数据。
8. 自动化、浏览器、迁移和生产只读验收全部通过。
